If such an address is pro- duced, the operation of the instruction generating such an address is UNPREDICTABLE. 4.11 32-bit Compatibility Addressing for Release 5 EVA, and Release 6 Section 4.10 “Special Behavior for Data References in User Mode with StatusUX = 0” applies to all architectural releases prior to Release 5, and Release 5 without Enhanced Virtual Addressing (EVA), where user mode data addresses are sign-extended only when Status UX =0. In Release 6, or Release 5 implementations that support EVA, the same sign-extension applies to all modes (Kernel/Supervisor/User) and both data and instruction addresses. Release 5 implementations with no support for Enhanced Virtual Addressing default to the behavior described in Section 4.10. Release 5 extensions of Status KX/SX/UX describe the new behavior. See CP0 Status definition for further detail. The example in Section 4.10 “Special Behavior for Data References in User Mode with StatusUX = 0” is also appli- cable to Release 5 with EVA except that the 32-bit base address of 0x8000_0000 may be a legal address, This is because the MIPS32 User Segment can exceed 2GB (from Release 5 onwards), which means a legal base address above 2GB can be generated by user software. The User Segment may be extended by virtue of “Enhanced Virtual Addressing (EVA)”, a MIPS32 feature introduced in Release 5. In this example, the resultant 64-bit address is cor- rectly placed in the lower 2GB 32-bit compatible virtual address range as a result of canonical sign-extension. Canon- ical sign-extension is the generation of a 64-bit address by sign-extension from bit 31. All Release 6 implementations must support the new addressing extensions for 32-bit compatibility. For all implementations supporting this behavior, the canonical check is performed on the input operands of the effec- tive address generation. The check is to prevent the accidental aliasing of non-canonical input addresses to a canoni- cal effective address, which can happen if, for example, kernel software uses a 64-bit instruction when Status KX =0. Kernel mode can always execute 64-bit instructions. If the input is non-canonical, then an Address Error must be sig- naled, either by the addressing instruction itself, or at the target of a control flow instruction. The Address Error exception is able to be taken at the target itself because the canonical error is propagated to the target address after address generation. The only exception for the control-flow case is Release 6 Jump Immediates (JIC/JIALC), where it is possible for the non-canonical base to be added to a sign-extended immediate and create a canonical address, which means the Address Error is undetected. This conversion of non-canonical to canonical happens if the resultant address crosses the boundary between the 32-bit compatibility regions - software must avoid such cases. The canonical sign-extension check of inputs could apply to up to 2 inputs of address generation in Release 5 or prior, which support indexed memory references. In Release 6, the check is limited to a single input as indexed addressing has been eliminated. The addressing extensions for 32-bit compatibility apply to : • An instruction fetch address. • An instruction such as a Load/Store, LL/SC, LLD/SCD, CACHE, PREF, SYNCI (and EVA extensions such as LWE).
